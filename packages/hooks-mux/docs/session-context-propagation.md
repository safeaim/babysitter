# Session Context Propagation

Deep dive on how the hooks-mux manages session state, context variables, and environment propagation across hook invocations and downstream tool execution.

---

## Context Categories

The proxy distinguishes three categories of context:

### 1. Persisted Environment Variables

Key-value pairs intended to appear in the environment of downstream shell/tool processes.

```typescript
// In a handler result:
return {
  persistEnv: {
    MY_API_TOKEN: 'abc123',
    MY_FEATURE_FLAG: 'true',
  },
};
```

These are stored in the session store and propagated via the adapter's env persistence backend.

### 2. Context Variables

Structured key-value metadata stored in the session store. Available to subsequent hook invocations but not automatically injected into subprocess environments.

```typescript
return {
  contextVars: {
    'my-plugin.iteration-count': '5',
    'my-plugin.last-tool': 'Bash',
  },
};
```

Access in later hooks via `event.execution.contextVars`.

### 3. Injected Model Context

Free-text or structured context sent back to the harness as additional developer/model context. Only available on adapters that support `additionalContext`.

```typescript
return {
  additionalContext: 'The user prefers TypeScript over JavaScript.',
};
```

## Session Lifecycle

### Creation

A session is created on the first hook invocation that resolves a session ID. The proxy:

1. Resolves the session ID from (in precedence order):
   - Explicit `--session-id` CLI flag
   - `AGENT_SESSION_ID` environment variable
   - `session_id` field in stdin payload
   - Adapter-specific resolution (e.g., Claude provides native session IDs)
2. Checks the session store for an existing session
3. Creates a new session if none exists

### Updates

Each hook invocation can update the session by returning `persistEnv`, `unsetEnv`, or `contextVars` in the result. The proxy merges these into the session state atomically.

### Resolution Quality

The `sessionIdQuality` indicates how the session ID was obtained:

| Quality | Meaning |
|---------|---------|
| `native` | Provided by the harness (e.g., Claude `session_id`) |
| `derived` | Computed from available signals (e.g., PID + timestamp) |
| `synthetic` | Generated by the proxy (no harness support) |
| `none` | No session ID available |

## Propagation Backends

### Mode A: Native Env File

The harness natively supports writing environment exports to a file that is sourced by later commands.

**Used by:** Claude Code (via `CLAUDE_ENV_FILE`)

```
# The proxy appends to CLAUDE_ENV_FILE:
export MY_VAR="value"
export ANOTHER_VAR="other"
```

### Mode B: Runtime Hook

The harness runtime natively supports injecting environment variables into all shell execution.

**Used by:** Pi, Oh-My-Pi, OpenCode, OpenClaw

The adapter communicates persisted env back through the hook result format, and the runtime injects them into subsequent tool processes.

### Mode C: Wrapper Execution

No native env injection. The proxy wraps commands to inject environment.

**Used by:** Codex, Gemini, Copilot, Cursor

Use the `exec` command to wrap downstream commands:

```bash
a5c-hooks-mux exec --session-id "$AGENT_SESSION_ID" -- npm test
```

This loads the session state, materializes persisted env into the subprocess environment, and then executes the target command.

### Mode D: Session Store Only

No downstream execution injection exists. Context is durable only inside proxy-managed hooks or explicit `exec` wrappers.

**Used by:** Custom observer-only adapters

## The `exec` Wrapper

For harnesses without native env persistence, use `exec` to inject session context:

```bash
# Basic usage
a5c-hooks-mux exec --session-id "$AGENT_SESSION_ID" -- npm test

# The proxy:
# 1. Loads session state for the given session ID
# 2. Materializes persistedEnv into the subprocess environment
# 3. Optionally generates a temp env file
# 4. Executes the target command with the enriched environment
```

### How It Works

1. **Load session** -- reads the session JSON from the session store
2. **Materialize env** -- converts `persistedEnv` into environment variable assignments
3. **Optional temp file** -- for tools that read env from a file, generates a temporary env file
4. **Exec** -- replaces the current process with the target command, carrying the enriched environment

## Session Store Structure

Sessions are stored as JSON files in the session directory:

```
~/.a5c/state/hooks/sessions/
  <session-id>.json
  <session-id>.json.lock
```

Each file contains:

```json
{
  "schemaVersion": "a5c.hooks.session.v1",
  "session": {
    "version": "a5c.hooks.session.v1",
    "sessionId": "abc-123",
    "adapter": "claude",
    "createdAt": "2026-04-17T10:00:00.000Z",
    "updatedAt": "2026-04-17T10:05:00.000Z",
    "cwd": "/home/user/project",
    "persistedEnv": {
      "AGENT_SESSION_ID": "abc-123",
      "MY_PLUGIN_VAR": "value"
    },
    "contextVars": {
      "my-plugin.counter": "3"
    },
    "contextFragments": [],
    "metadata": {}
  }
}
```

### Atomic Writes

The session store uses the same atomic write protocol as the babysitter SDK:
1. Write to a temp file (`<path>.tmp-<pid>-<timestamp>`)
2. `fsync` the temp file
3. Rename to the target path

### Locking

Concurrent access is protected by file locks (`.lock` files) using an exclusive-open (`wx`) strategy with retry.

### Platform-Specific Paths

| Platform | Default Path |
|----------|-------------|
| Linux/macOS | `${XDG_STATE_HOME:-~/.local/state}/a5c-hooks/sessions/` |
| Windows | `~/.a5c/state/hooks/sessions/` |

Override with environment variable or CLI flag.

## Multi-Plugin Merge Semantics

When multiple handlers execute during fan-out, their context outputs are merged:

### Env Merge Rules

- **`persistEnv`**: Key-wise merge. Last writer wins by default unless the key has a protected prefix.
- **`unsetEnv`**: Union of all unset keys.
- **`contextVars`**: Key-wise merge. Last writer wins by default.

### Conflict Policies

| Policy | Behavior |
|--------|----------|
| `last-writer-wins` | Later handler overwrites earlier (default) |
| `fail-on-conflict` | Error if two handlers write different values to the same key |
| `protected-prefixes` | Keys with protected prefixes (e.g., `AGENT_`) cannot be overwritten |
| `namespace-required` | All keys must have a namespace prefix |

### Reserved Prefixes

- `AGENT_*` -- reserved for proxy internals
- Plugin-specific keys should use a namespace: `PLUGIN_X_*`

### Capabilities Propagation

The proxy injects `AGENT_CAPABILITIES_JSON` into handler subprocess environments and materialized exec contexts. This contains the full `AdapterCapabilities` object serialized as JSON, allowing downstream consumers to inspect what the originating harness supports:

```typescript
import { readExecutionContext } from '@a5c-ai/hooks-mux-core';

const ctx = readExecutionContext();
if (ctx.capabilities) {
  console.log(ctx.capabilities.supportsBlock);       // true/false
  console.log(ctx.capabilities.envPersistenceMode);  // 'native_env_file' | 'runtime_hook' | ...
  console.log(ctx.capabilities.toolInterceptionScope); // 'all' | 'shell_only' | ...
}
```

The variable is set in three places:
1. **Shell handler subprocess env** (`runner.ts`) -- injected alongside other `AGENT_*` vars
2. **Materialized exec context** (`materialize.ts`) -- available for `exec` wrapper commands
3. **Programmatic engine event metadata** (`engine.ts`) -- accessible as `event.execution.metadata['AGENT_CAPABILITIES_JSON']`

## Diagnostics

The proxy emits structured diagnostics for every hook execution, including:

- Session ID quality and resolution method
- Which handlers executed and in what order
- Merge decisions and conflicts
- Output degradation flags (fields that could not be faithfully represented in native output)

Enable JSONL trace files for full execution traces:

```bash
AGENT_HOOKS_TRACE_FILE=./hooks-trace.jsonl a5c-hooks-mux invoke --adapter claude --native-event SessionStart
```

## Troubleshooting

### Session state not propagating

1. Check the adapter's env persistence mode: `a5c-hooks-mux doctor --adapter <name>`
2. If mode is `wrapper_only` or `none`, use the `exec` wrapper for downstream commands
3. Verify `AGENT_SESSION_ID` is available in the environment

### Stale sessions accumulating

Run the doctor to identify stale sessions:

```bash
a5c-hooks-mux doctor --stale-threshold 24
```

Clear individual sessions:

```bash
a5c-hooks-mux clear-session --session-id <id>
```

### Context conflicts between plugins

1. Namespace all env keys with a plugin-specific prefix
2. Use `fail-on-conflict` policy during development to detect collisions
3. Check merge diagnostics in the structured log output
